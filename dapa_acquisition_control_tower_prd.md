# PRD: DAPA 획득체계 통합관제 서비스

## 1. 제품 개요

### 제품명

`DAPA 획득체계 통합관제 서비스`

영문명: `DAPA Acquisition Control Tower`

### 한 줄 설명

방위사업청 공공데이터를 활용해 무기체계 획득사업의 조달계획, 입찰공고, 입찰결과, 계약정보를 통합 분석하고, AI가 유찰·지연·경쟁 부족·비용 괴리·이상 낙찰 위험을 조기에 탐지해 담당자에게 실행 가능한 의사결정 브리핑을 제공하는 서비스

### 핵심 문제

방위력개선사업은 사업 규모가 크고, 조달·계약·일정·예산·규정 준수 리스크가 복합적으로 발생한다. 그러나 현재 실무자는 여러 시스템과 파일을 따로 확인하고, 과거 유사사례를 경험에 의존해 찾으며, 엑셀로 공고·입찰결과·계약정보를 직접 비교하는 방식에 의존한다.

이 서비스는 경험 기반 판단을 데이터 기반 판단으로 전환하는 것을 목표로 한다.

## 2. 조합한 아이디어

`DAPA 획득체계 통합관제 서비스`는 기존 10개 아이디어 중 4개를 1차 MVP로 조합한다.

| 조합 아이디어 | 서비스 내 역할 | MVP 포함 여부 |
| --- | --- | --- |
| DAPA Acquisition Risk Radar | 유찰, 지연, 경쟁 부족 위험 예측 | 포함 |
| Defense Cost & Schedule Early Warning | 예산 대비 계약금액, 발주-공고-계약 일정 편차 감지 | 포함 |
| RFP Quality & Competition Optimizer | 입찰공고 조건과 과거 결과를 비교해 경쟁 부족 가능성 점검 | 포함 |
| Bid Fairness & Anomaly Detector | 낙찰률, 반복 낙찰, 단일 참여 등 이상패턴 탐지 | 포함 |
| Defense Decision Briefing Agent | 사업별 1페이지 의사결정 브리핑 자동 생성 | 2차 확장 |
| Defense Compliance Copilot | 법·규정 준수 체크 | 2차 확장 |
| K-Defense Export Opportunity Matcher | 해외입찰·수출 기회 매칭 | 별도 서비스 또는 3차 확장 |

1차 MVP는 `획득·조달·계약 리스크 관제`에 집중한다. 수출, 법령 RAG, 장병 신체 데이터 기반 장비 수요 예측은 확장 기능으로 분리한다.

## 3. 사용자와 업무 시나리오

### 주요 사용자

| 사용자 | 현재 불편 | 서비스 제공 가치 |
| --- | --- | --- |
| 사업관리 담당자 | 과거 유사사업, 지연 사례, 계약 결과를 수작업 검색 | 사업별 리스크 점수와 유사사례 자동 제공 |
| 계약 담당자 | 공고, 입찰결과, 계약금액, 낙찰률을 엑셀로 비교 | 계약방법별 성과와 이상징후 자동 분석 |
| 정책·기획 담당자 | 품목별 조달 추세와 병목을 정량적으로 보기 어려움 | 품목·부서·기관별 리스크 대시보드 제공 |
| 감사·점검 담당자 | 일부 표본을 수작업으로 점검 | 전수 기반 이상패턴 후보 목록 제공 |
| 신규 담당자 | 선배 경험과 내부 관행에 의존 | 데이터 기반 과거사례와 판단 근거 제공 |

### 대표 시나리오

1. 담당자가 특정 사업명 또는 품목명을 검색한다.
2. 시스템이 조달계획, 입찰공고, 입찰결과, 계약정보를 연결한다.
3. 사업 타임라인을 `계획 -> 공고 -> 개찰 -> 계약` 순서로 보여준다.
4. AI가 유찰 위험, 일정 지연 위험, 경쟁 부족 위험, 비용 괴리 위험을 점수화한다.
5. 시스템이 유사 과거사업 5건과 차이점을 보여준다.
6. 담당자는 고위험 원인과 조치 권고를 확인하고 회의자료로 활용한다.

## 4. 현 상황 및 추진배경

### 현 상황

- 조달계획, 입찰공고, 입찰결과, 계약정보가 각각 다른 데이터셋으로 존재한다.
- 담당자는 사업명, 품목명, 공고번호, 계약명을 기준으로 수작업 연결을 수행한다.
- 공고 후 유찰 가능성, 경쟁 부족 가능성, 계약 지연 가능성을 사전에 정량화하기 어렵다.
- 낙찰률이나 계약금액이 유사 사업 대비 특이한지 판단하려면 별도 엑셀 분석이 필요하다.
- 신규 담당자는 과거 사업의 시행착오와 유사사례를 빠르게 찾기 어렵다.

### 추진배경

방위사업은 실패 비용이 크다. 일반 행정서비스보다 일정 지연, 계약 리스크, 업체 경쟁 부족, 공급망 집중, 법·규정 미준수의 파급효과가 크다. 따라서 공개 공공데이터만으로도 먼저 `조기경보 체계`를 구축하고, 이후 내부 데이터와 연계해 의사결정 품질을 높이는 단계적 접근이 필요하다.

### 5.3 데이터 수집 실행 방식

#### OpenAPI 수집

1. 공공데이터포털 회원가입 및 각 OpenAPI 활용신청
2. 서비스키 발급
3. 최근 3~5년 범위를 월 단위로 나눠 호출
4. `numOfRows`, `pageNo`를 사용해 페이지네이션 처리
5. 원천 응답 JSON/XML을 그대로 저장
6. 정규화 테이블로 변환

예시 수집 단위:

```text
조달계획: 발주예정월 202101~202612 월 단위
입찰공고: 공고일자 20210101~20261231 월 단위
입찰결과: 개찰일자 20210101~20261231 월 단위
계약정보: 계약일자 20210101~20261231 월 단위
```

#### CSV 수집

1. 공공데이터포털에서 파일데이터 다운로드
2. 데이터 갱신일 기준으로 버전 관리
3. UTF-8 또는 CP949 인코딩 확인 후 적재
4. 컬럼명 표준화
5. 업체명, 계약명, 공고명 정규화

### 5.4 데이터 연결 전략

공공데이터만으로 모든 단계가 완벽한 단일 키로 연결된다고 전제하면 안 된다. 따라서 정확 매칭과 유사 매칭을 함께 사용한다.

| 연결 구간 | 1순위 키 | 2순위 키 | 보완 전략 |
| --- | --- | --- | --- |
| 조달계획 -> 입찰공고 | 판단번호, 대표품목명, 발주기관, 요구년도 | 품목명 유사도, 발주예정월-공고일자 차이 | 품목명 임베딩 + 날짜 윈도우 |
| 입찰공고 -> 입찰결과 | 입찰공고번호, 공고차수, G2B공고번호차수 | 입찰명, 발주기관, 개찰일자 | 정확 매칭 우선, 실패 시 유사도 점수 |
| 입찰결과 -> 계약정보 | 공고번호, 계약명, 업체명 | 입찰명-계약명 유사도, 개찰일자-계약일자 차이 | 금액·날짜·기관 기준 검증 |
| 계약정보 -> 업체정보 | 계약업체명, 업체명 | 업체명 정규화 | 법인명 접미어 제거, 공백·특수문자 제거 |

연결 결과에는 반드시 `match_confidence`를 둔다.

- 0.95 이상: 동일 키 기반 확정 연결
- 0.80~0.95: 유사도 기반 후보 연결
- 0.80 미만: 수동 검토 필요

## 6. 핵심 기능 요구사항

### F1. 통합 사업 타임라인

사업명 또는 품목명을 입력하면 다음 흐름을 한 화면에 표시한다.

```text
조달계획 -> 입찰공고 -> 입찰결과 -> 계약정보
```

표시 항목:

- 발주예정월
- 공고일자
- 입찰마감일
- 개찰일자
- 계약체결일자
- 계약기간
- 진행상태
- 계약금액

### F2. 리스크 점수

사업 또는 입찰 건별로 5개 점수를 산출한다.

| 리스크 | 산출 방식 |
| --- | --- |
| 유찰 위험 | 유사 품목·계약방법·기관의 과거 유찰률, 마감기간, 참여업체 수 |
| 경쟁 부족 위험 | 과거 동일·유사 품목 평균 참여업체 수, 단일 참여 여부 |
| 일정 지연 위험 | 발주예정월 대비 공고일자, 공고일자 대비 계약일자 편차 |
| 비용 괴리 위험 | 예산금액, 예정가격, 계약금액, 낙찰률의 유사군 대비 편차 |
| 이상 낙찰 위험 | 낙찰률, 반복 낙찰, 동일 업체 집중도, 수의계약 사유 패턴 |

### F3. 유사사업 검색

사용자가 사업명 또는 품목명을 입력하면 유사 과거사례를 추천한다.

유사도 기준:

- 품목명 텍스트 유사도
- 계약방법
- 입찰방법
- 발주기관
- 예산 규모
- 업무구분
- 계약기간

### F4. 이상징후 탐지

다음 조건을 자동 탐지한다.

- 유사 품목 대비 낙찰률 과소·과대
- 동일 업체 반복 낙찰
- 단일 업체 참여 또는 참여업체 급감
- 제한경쟁·수의계약 집중
- 발주예정월 대비 공고 지연
- 계약기간이 유사 계약 대비 과도하게 긴 경우

### F5. 공고 품질 점검

입찰공고 데이터를 기준으로 경쟁성을 저해할 가능성이 있는 조건을 점검한다.

1차 MVP에서는 공고문 전문 분석보다 정형 데이터 기반 점검에 집중한다.

- 마감기간이 유사 공고 대비 짧은가?
- 계약방법이 유사 품목 대비 제한적인가?
- 과거 동일 품목에서 반복 유찰이 있었는가?
- 참여업체 수가 적었던 과거 조건과 유사한가?

### F6. 의사결정 브리핑

사업별 1페이지 요약을 생성한다.

포함 항목:

- 현재 상태
- 주요 리스크 점수
- 위험 원인
- 유사 과거사례
- 담당자 확인 필요 항목
- 추천 조치
- 데이터 출처 및 기준일

## 7. AI 적용 방식

### 7.1 초기 MVP

초기에는 복잡한 딥러닝보다 설명 가능한 모델과 규칙 기반 모델을 우선한다.

| 기능 | 적용 기술 | 이유 |
| --- | --- | --- |
| 유찰·경쟁 부족 예측 | Logistic Regression, Random Forest, XGBoost | 정형 데이터 예측에 적합 |
| 비용·일정 이상탐지 | 통계 기준선, Isolation Forest | 설명 가능성과 구현 용이성 |
| 유사사업 검색 | TF-IDF/BM25 + 임베딩 유사도 | 품목명·사업명 변형 대응 |
| 브리핑 생성 | LLM 구조화 출력 | 담당자 보고서 초안 작성 지원 |

### 7.2 AI 결과 표시 원칙

## 12. 실행 가능성 평가

### 매우 높은 부분

- 데이터 접근: 조달계획, 입찰공고, 입찰결과, 계약정보는 공공데이터포털 OpenAPI로 제공된다.
- 활용 승인: 개발단계 자동승인으로 명시된 데이터가 있어 MVP 수집이 가능하다.
- 비용: 무료 데이터이며 이용허락범위 제한 없음으로 표시되어 있다.
- 분석 대상: 개인정보가 아닌 조달·계약 중심 데이터라 공모전 프로토타입에 적합하다.
- 초기 모델: 정형 데이터 기반 예측과 이상탐지는 공개데이터만으로 구현 가능하다.

### 주의가 필요한 부분

- 데이터셋 간 단일 연결키가 부족할 수 있다.
- 계약명, 입찰명, 품목명이 완전히 일치하지 않을 수 있다.
- 참여업체 수나 낙찰업체 정보는 상세 API 또는 별도 파일에서 보강해야 할 수 있다.
- 공고문 전문 분석은 공공데이터에 포함된 범위에 따라 제한될 수 있다.
- 실제 정책 판단에는 내부 데이터와 담당자 검토가 필요하다.

### 대응 방안

- 정확 키 연결과 유사도 연결을 병행한다.
- 연결 신뢰도 점수를 표시한다.
- MVP에서는 자동 판정이 아니라 `검토 우선순위 추천`으로 범위를 제한한다.
- 공고문 전문이 부족하면 정형 항목 기반 공고 품질 점검부터 시작한다.
- 내부 데이터 연계는 향후 계획으로 제시한다.

## 14. 공모전 제출 메시지

`DAPA 획득체계 통합관제 서비스`는 방위사업청이 보유·개방한 조달계획, 입찰공고, 입찰결과, 계약정보를 연결해 기존에 직원 경험과 엑셀 수작업에 의존하던 획득 리스크 판단을 데이터 기반으로 전환한다. AI는 유찰, 지연, 경쟁 부족, 비용 괴리, 이상 낙찰 가능성을 조기에 탐지하고, 담당자가 검토해야 할 근거와 유사 과거사례를 함께 제시한다.

이 서비스는 실제 의사결정을 대신하지 않는다. 대신 담당자가 더 빠르고 정확하게 판단할 수 있도록 `어떤 사업을 먼저 봐야 하는지`, `왜 위험한지`, `어떤 과거사례와 유사한지`, `어떤 조치를 검토해야 하는지`를 한 화면에서 제공한다.
